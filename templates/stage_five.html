Add the JavaScript inside of <script> tags. This example shows how.

var loveJS = true;

JavaScript runs natively in the browser and allows us to run and inspect our code very easily.

JavaScript gives us a way of saving data in the form of variables.

The syntax looks like this: var {{variableName}} = {{someValue}}.

To declare a new variable, you begin with the keyword var, regardless of the datatype you're saving.

In this example, var firstName = "james";, we're creating a variable, firstName, and assigning it the string "james".

Assigning variables is slightly different than in Python, which doesn't require a keyword like var, but otherwise looks the same.

Numbers in JS don't need quotes, and you don't need to worry about the differences between floating point numbers and integers, as all numbers are saved as 64 bit floating point.

Later on, we'll see that arrays, functions and objects use the same var syntax.

Lets see if our variables are actually being saved. Add these lines to resumeBuilder.js:

If the browser opens up a script with console.log in it, it will print whatever information it recieves to the browser console. Its very useful for debugging.

Dot and Bracket Notation

We just used dot notation for objects, which we just used to access bio.name.

Dot notation allows us to define an object, then define its properties outside the curly braces:

Syntax: var myObj = {};

myObj.property = someValue;

Example:

var bio = {
	"name" : "James",
	"age" : 32
	"skills" : skills
};
bio.city = "Mountain View";
bio.email = "james.williams@udacity.com"

With dot notation, there is no need to use var. We're adding properties to an object, not creating new variables.

There is at least one more syntax pattern to define the properties of an object: Bracket notation.

Syntax: myObj["property"]=someValue

Example:

bio ["city"] = "Mountain View";
bio ["email"] = "james.williams@udacity.com"
A Note about JSON

What is JSON?

JavaScript Object Notation. Its a simple format for storing and transferring nested hierarchical data. Most other programming languages have libraries to parse and write JSON. GET and POST requests often communicate using JSON. JSON allows for objects and data of other types to be enclosed in other objects.

Why should I lint my JSON?

JSON uses a mix of nested curly braces, brackets and commas, so its easy to make errors, creating bugs. If you're writing JSON by hand, you should run your code through a JSON linter to find syntax errors. jsonlint.com is a good one.

Sublime text will highlight most syntax errors, but a JSON linter won't miss any.

Stage 5.2 The Power of APIs

Introduction

We're going to talk about how to make your website communicate with other computers, by using what we call an API (Application Programming Interface.) Instead of generating HTML that we send to browsers, we're going to get data in formats like XML or JSON which are designed to be read by computers. In this lesson, we're going to add a feature to ASCII Chan that will allow you to see where users are submitting images from using the Google Maps API.

HTTP Clients

Until now, we've been working with the standard picture of the user, their browser, and servers. In a normal web request, the user makes a request to the server, and we respond with the response. What we're going to be talking about is when your servers make requests to other servers. We can have a web page that actually makes requests to Twitter. These are our servers talking to theirs. They're still communicating over HTTP, and Twitter still responds as usual. If we're writing a web program that does some data analysis on Twitter, a user might make a request to us, we might make a request to Twitter, they respond, we may manipulate that data and return it to the user. This is actually a really common case.

Hipmunk does a lot of this. When a user does a flight search, what Hipmunk does is hit a bunch of data providers where they actually get their flight data from. Hipmunk will take your flight search, send it to ITA, send it to an airline, and in some cases, even send it to Amtrak, ITA will run their flight search and send Hipmunk data back, the airline will do the same, and Amtrak will do the same as well. On Hipmunk's server, they have all this flight search data. They will manipulate all the data, collate it, make nice results, and send back an HTML response to the user.

What we're going to be working on is how do we make our server speak to other servers, when there's no browser involved. We're still using HTTP, but we are now communicating over other protocols. There are a lot of cool things you can do when you realize you aren't the only service on the internet.

Urllib

In Python we have a library called urllib2. (And a library called urllib.) Open a Python terminal and import both of those.

urllib2 has a function called urlopen. Type this line into the terminal: p = urllib2.urlopen("http://www.google.com")

Now if we type p in the terminal, we get something like: <addinfourl at 42571208L whose fp = <socket._fileobject object at 0x0000000002466B10>>

This is basically a file object. In Python, a file object is an object with a read method. You can call read on it to get the contents.

Type this line into the terminal: c = p.read()

Now, type in c. We get a wall of text. This is actually Google's front page. Now we can manipulate the response in our programs.

Now, lets look at the p object. Type in dir(p). Now we can see the methods and attributes on our p object.

Lets look at a couple of them: Type in p.url. We get 'http://www.google.com'.

Type in p.headers. We get something like <httplib.HTTPMessage instance at 0x0000000002899288>. This is an HTTP message instance. This happens to be a dictionary. So if we type in p.headers.items(), we can see all the headers we got back from Google.

XML

How computers communicate:

We could have our servers (say Hipmunk,) make a request to Amtrak and receive HTML back from them. But, this is sub-optimal. HTML is somewhat complex, its not very regular, browsers are forgiving of errors, and parsing HTML by machine is very difficult, so HTML is not an ideal language for computer to computer communication.

One such language is XML. XML was invented in the late 90s expressly for this purpose, to have a regular way of expressing data between computer systems. XML is very similar to HTML. The main difference between XML and HTML is that every tag must have a closing tag. The tag format is the same, we still have less than and greater than surrounding tags, and slashes to indicate closing tags. There are no void tags in XML.

Parsing XML

Python has a library called minidom. You can get it by saying from xml.dom import minidom. dom stands for Document Object Model. This refers to the internal representation of an XML document. We'll be using minidom. Why is it called that? The mini implies that its a smaller, lighter weight version of this dom parser. When you're only parsing a little text, you can use the library minidom, which is simple and fast, but will break if you throw lots of gigabytes of text at it.

minidom has a function called parseString: minidom.parseString

Try entering this: x = minidom.parseString("<mytag>contents!<children><item>1</item><item>2</item></children></mytag>")

Then do dir(x). We get a ton of stuff back.

Try print x.toprettyxml(). We get a nicely formatted version of our string.

Try x.getElementsByTagName("mytag"). We get one item.

Try x.getElementsByTagName("item"). We get two items.

We can look at the first item element by doing x.getElementsByTagName("item")[0]

If we look at its children, x.getElementsByTagName("item")[0].childNodes, we get [<DOM Text node "u'1'">], which means we have one text node.

x.getElementsByTagName("item")[0].childNodes[0].nodeValue accesses the node value attribute, which tells us the value is 1.

JSON

JSON serves the same purpose as XML, which is that its a nice computer and human readable way to exchange data. It stands for JavaScript Object Notation. JSON is actually valid JavaScript code. It basically looks like a dictionary:

{"itineraries": [{"from": "SFO",
				  "to": "IAD"},
				 {"from": "IAD",
				  "to": "SEA"}]
}

We have a dictionary, its got a key called itineraries, whose value may be a list of other dictionaries. In this case, we have a dictionary for each leg.

JSON is really handy for expressing these types of objects. Anything you can express in XML, you can also express in JSON. JSON is a little less verbose because you don't need opening and closing tags. You can build things out of dictionaries, you can also have lists, which use brackets just like Python does, a list can also be the value of a dictionary, or an item in a list.

How to parse JSON in Python:

In your Python terminal import json.

Enter this: j = '{"one": 1, "numbers": [1,2,3.5]}'

Now, lets parse the string: json.loads(j)

When we run that, we get back a Python dictionary with our same keys and our same values: {u'numbers': [1, 2, 3.5], u'one': 1}

If we store that in a variable like this: d = json.loads(j)

We can manipulate it like this: d['numbers'] gives us our list, and d['one'] gives us our number 1.

APIs

On Reddit, we can get any page in JSON or XML by changing the extension to .json or .xml respectively. Many web pages have this feature, where you can get their content in different formats.

Another good example is Twitter. If you go to Twitter and do a search for Udacity, you'll get all the different tweets about Udacity. Instead, if you go to search.twitter.com/seach.json?=udacity, you'll get a json listing of the search results we were just looking at.

If you're writing web software and want to include data from other websites, it usually just takes some poking around to find their APIs.

Being a Good Citizen

How to be a good citizen on the internet:

There are two key things you can do when writing software that accesses other people's websites.

1. Use a good user-agent. User-agents are the header that describes the browser or software you're using to access somebody. If you're planning to access somebody in a consistent fashion, use a good user agent. When you're using urllib2, you can specify headers in your request, and you should set a user agent header that says who you are, what your name is, maybe links to your website, so if someone on the other end sees you pounding them, they have a way of reaching you to ask you to tone it down.

2. Rate limit yourself. Instead of hammering sites for more and more results, its a good idea to get in the habit of using the sleep function so you aren't stressing someone's server.

Debugging

Lets test to see if our code is working. Add this line to def get before it calls render_front():

self.write(repr(get_coords(self.request.remote_addr)))

First we call the function write, which outputs to the browser. Then we call repr. This is a handy trick for printing Python objects in HTML. When you print Python objects in HTML, it has angle brackets around them. repr makes them print properly. Then we call our function, get_coords, and call it with the requesting ip address (self.request.remote_addr).

When we save our code and reload the page, we see None at the top of the page. Thats better than an exception, but we didn't get a location.

Lets do some investigation: First lets look at the actual ip address:

	def get(self):
		self.write(self.request.remote_addr)
		self.write(repr(get_coords(self.request.remote_addr)))
		return self.render_front()

See what the browser returns now. 127.0.0.1. That means we're running this locally, so its not surprising that a service on the internet doesn't know what our local ip is. If we go to the hostip.info web page, it lists 127.0.0.1 as a private address. Every machine's local ip address is 127.0.0.1. This is called the loopback address and its how a machine refers to itself. Its not a public ip address on the internet. Localhost, which we're running our webpage on, generally refers to that ip.

Stage 5.3 Recursion and Parallel Computing

Infinite Power

The big idea in this stage is recursive definitions. The real goal is to give you infinite power. But wait, we've had infinite power since stage 2, when we learned that with if, functions and arithmetic we could every possible computer program. We went on to introduce while, and that really should not have been necessary if you already had infinite power. What we're going to see in this unit is all you really need are procedures (functions) and that by thinking about things recursively you can solve lots of problems in interesting ways.

Long Words

What is the longest word in the English language?

Is it hornorificabilititudinitatibus, antidisestablismentarianism, hippopotomonstrosequipedaliophobia pneumonoultramicrosopicsilicovolanoconiosis, or none of the above? The answer is none of the above. The reason is that whatever you claim is the longest word, you can always make a longer one. There is no such thing as the longest word in the English language.

Counter

A word is just something that has meaning, that speakers of the language will understand. We could define words as just what's in the dictionary and then there would be a clear answer to the longest word question. But, there are lots of things that are words that aren't in the dictionary. The important thing about words is we have rules for making new words.

For example, we have a rule that says if we have a word, we can make a new word by adding counter- in front of the old word:

Word --> counter-Word. This is a Backus-Naur Form replacement grammar that we used in stage 2. The basic property is that we can replace what's on the left side with what's on the right side.

Any time we have a word, we can replace it with counter-word, the meaning of the new word is "something that goes against the original word."

If we started with the word intellegence, (in the sense of spycraft,) we could use the rule and make the word counter-intelligence. We can use the rule again and make counter-counter-intelligence. We could keep doing that, we could have counter-counter-counter-intelligence, and so on.

We could do this for other words, like hippopotomonstrosequipedaliophobia (fear of long words.) We can make the new word counter-hippopotomonstrosequipedaliophobia, and even though you've never seen the word before, you can guess that its something that goes against the fear of long words, like a new drug. You could also guess that counter-counter-hippopotomonstrosequipedaliophobia is something that counteracts the new drug.

Because English has this rule that allows us to make new words from old words, there is no such thing as the longest word. We can always make a longer word starting from any word that we have.

Counter Quiz

If the only rule we have for making words is this one:

Word --> counter-Word

How many words can we make, starting from the non-terminal Word?

None, 1, 2, or Infinitely many?

The answer is none.

We can't make any words from this rule, because our definition is circular. If you remember BNF grammars, we can only stop once we have all terminals. This rule gives us no way to stop. We can keep adding more counter-s but, we'll never replace the non-terminal Word.

Expanding Our Grammar

Lets add one more rule.

How many different words can we make starting from Word using only these two rules:

Word --> counter-Word

Word --> hippopotomonstrosequipedaliophobia

How many words can we make?

None, 1, 2, or Infinitely many?

Now, the answer is infinitely many.

This is the power that recursive definitions give us.

Unlike the previous definition which was circular, what we have now is called a recursive definition. That means we've defined Word in terms of itself, but we have another rule that allows us to have a starting point. Using these two rules, there is no limit to the number of words we can produce.

The important things that our definition has is a base case: Word --> hippopotomonstrosequipedaliophobia

And, it has a recursive case: Word --> counter-Word

By combining those two, we have a definition that can make infinitely many words.

Recursive Definitions

This works for things other than words. The place where you've seen it the most so far has been in language. We're also going to see how to use it to make procedures. There are lots and lots of different things in computing that are defined recursively.

A recursive definition has two parts:

1. The base case. A starting point. The important thing about the base case is that its not defined in terms of itself. Its the smallest input that we already know the answer to.

2. The recursive case. It is defined in terms of itself. Actually, defined in terms of a smaller version of itself. We need to make progress to eventually get to the base case.

Ancestors

Who are your ancestors?

We have you, and your parents. But they aren't all of your ancestors. Your parents have parents as well. Your grandparents also had parents, your great-grandparents, and so on.

All of these, except yourself, are your ancestors.

Our goal is to define that precisely. Can we come up with a definition that describes preciesly all the ancestors that you have?

The best definition of ancestors is:

Ancestor --> Parent

Ancestor --> Parent of Ancestor

This is a recursive definition that works. It says our parent is our ancestor, and the parent of an ancestor is also an ancestor. This will give us exactly the set of ancestors that we talked about.

Recursive Procedures

Now we're going to see how to use recursive definitions to define a procedure.

We're going to start by defining a factorial procedure. According to Wikipedia, the factorial of a non-negative integer n, denoted by n!, is the product of all positive integers less than or equal to n.

factorial (n) = n * (n-1) * (n-2) * ... *1

This definition is not very precise mathematically. The problem is the ... . Humans understand the ... correctly, but its not a very precise mathematical definition.

If we use a recursive definition, we can define factorial in a much more precise way.

We need to do that by giving a base case: For the base case, we want to think about the simplest input, something we already know the answer to. For factorial, the simplest input is the number zero. factorial (0) = 1. Thats going to be our base case.

Now we need to define the recursive case: factorial (n) = n * factorial (n-1)

Recursive Factorial

Define a procedure, factorial, that takes a natural number as its input, and outputs the number of ways to arrange the input number of items.

def factorial(n):
	if (n) == 0:
		return 1
	else:
		return n * factorial(n-1)

We need to first check if we've reached the base case, so we use an if for that:

	if (n) == 0:
		return 1

When n is not equal to 0, we use else: Then we have the recursive case:

	else:
		return n * factorial(n-1)

It may seem strange that we're defining factorial using factorial, it seems circular. The reason its not circular is we have our base case. We have a case where we do stop. Every time we call factorial, instead of passing in the same value of n that we started with, we're passing in n-1, eventually we're going to get to 0. We eventually stop, return 1, and on the way there, we're going to be multiplying in all these values.

Lets step through what happens when we run this code. Suppose we called factorial(3). We're going to enter the procedure, the value of n is going to be 3. n is not equal to 0, so we return the value of n * factorial(n-1), so we're computing 3 * factorial(2). Now we're evaluating factorial(2). We enter the procedure again, this time the value of n is 2. We do the comparison, n does not equal 0, so we go to the else clause and return 2 * factorial(1). We call factorial again with factorial(1). We reenter factorial, but this time the value of n is 1. The if test is false, so now we're going to return the result of 1 * factorial(0). Now we call factorial(0), now the value of n is 0. The test is true, so we return 1. That gives us the solution to all our return statements, which is 6.

Palindromes

We're going to do another example of defining a recursive procedure. The goal this time is to take in a string and determine whether or not its a palindrome. A palindrome reads the same way forwards and backwards. An example of a palindrome would be the word level. If we read it forwards, we get level, if we read it backwards, we get exactly the same thing. Another example would be the word a. If we read it forwards or backwards, we get the same result. The empty string, "" is also a palindrome.

Our goal is to define a procedure, that will take any string as input, and output True if that string is a palindrome.

Define a procedure, is_palindrome, that takes as input a string, and outputs a boolean indicating if the input string is a palindrome.

Some hints: Define what it means to be a palindrome. We should think about the base case: "" -->True. What if the input is not an empty string? One way to solve that would be to look at the first letter and last letter of the string. If those are equal, it might be a palindrome, its a palindrome if all the letters left over in the middle are also a palindrome. So, here is how we can break the question of whether a string is a palindrome into smaller steps: Our recursive case is going to test the first and last character of the string, and see if those are equal, if the first and last characters don't match, we know the result is false. If they do match, we need to check to see if the middle of the string is a palindrome.

def is_palindrome(s):
	if s == '':
		return True
	else:
		if s[0] == s[-1]:
			return is_palindrome(s[1:-1])
		else:
			return False
MapReduce

According to Wikipedia, MapReduce is a programming model and an associated implementation for processing and generating large data sets with a parallel, distributed algorithm on a cluster.

Say you have a large file. Processing it serially could take a long time. Instead, MapReduce is designed to process data in parallel. So, your file is broken into chunks, and then processed in parallel. Lets look at a real world scenario.

Real World Example

Lets imagine we run a retailer, with thousands of stores around the world, and we have a big ledger that contains all of our sales. The Big Book O' Sales for the year 2012. We've been asked to calculate things like the total sales per store.

Lets say the lines of the ledger look like this: The date of the purchase, the location of the store, what they bought, and the amount.

One way we could do this is start at the beginning of the ledger, and for every entry, write down the store location and the amount. When we come to a store location again, update the amount by addition. Then we'd continue through the ledger.

Distributed Work

Rather than one person reading the entire ledger, say we had more people to help. We'll split them into two groups: The Mappers, and The Reducers.

Then we'll take the ledger, break it into chunks and give each chunk to one of the mappers. That way each of the mappers can work at the same time, each over a small fraction of the data. The mappers would take each sale, and write the location of the store and the amount on an index card. As they write the index cards, they'll pile them up so cards from the same store are in the same pile. By the end, each mapper will have a pile of cards per store.

Once the mappers have finished, the reducers can collect their sets of cards. We'll tell each reducer which store they're responsible for. The reducers go to the mappers and collect their respective piles of cards. Its fast because the mappers have already separated their cards. They amalgamate these piles. All they have to do at this point is add up all the amounts from all the cards in the pile. That gives them the total sales for that store. They can then write that amount on their final sheet. To keep things organized, each reducer goes through their piles in alphabetical order, in other words Los Angeles before Miami.

Summary of MapReduce

Thats how MapReduce works.

The mappers are just little programs that each deal with a relatively small amount of data and work in parallel. We call their output intermediate records. Thats what we were writing on our index cards. It deals with all data in terms of (key, value), so these records are actually keys and values. In our example, the key was the store name and the value was the sale total for each particular piece of input.

Once the mappers have finished, a phase of MapReduce called shuffle and sort takes place. The shuffle is the movement of the intermediate records from the mappers to the reducers. The sort is the fact that the reducers will organize the sets of records, in our case the piles of cards, into sorted order.

Finally, each reducer works on one set of records at a time, or one pile of cards. It gets the key and then a list of all the values. In our case a store like New York City, and all the sales for that store. The reducer processes those values in some way, in our case we were adding up sales. Then it writes out the final results.

Solving Big Problems

Peter uses an approach that we learned in Stage 1, Code, Test & Refine and in Stage 2 How to Solve Problems, but adapts it to solving a large problem.

So, here is my take on Peter's process:

First, he takes time to fully comprehend the problem and break it down into smaller parts, which he calls a "concept inventory":

board

letters

--words

--hand

legal play

score

--letters

--play

--bonus

dictionary

blank tiles

Second, he goes through his list and figures out basically how to implement each of the ideas in the inventory, seeing if there are any difficulties or areas that might be hard to implement.

board - 2d array, may need a 2nd board to account for bonus squares

letters - str

--words - str

--hand - str

legal play - function, plays will need a position, direction, and the word itself

score - function

--letters - dictionary {'Z': 10}

--play - function

--bonus - mapping of position on the board to DW, etc.

dictionary - set(words)

blank tiles - str, dealing with it will be an issue later. Not only do we have to try all possiblities for the letter, the scoring rules are different.

Third, he assures himself that solving the problem is doable. He doesn't quite know how each solution will work, but nothing he sees is unsolvable.

Fourth, he reminds himself and the audience of the notion of pacing. He isn't going to sit down for 20 minutes and solve the problem. Its going to have to be incrementally, where he sets intermediate goals along the way. Without that pacing, you can lose focus and get discouraged that there is so much to do. If you break the problem into bite-sized pieces, you can say "I'm almost there" and move on to the next part

Fifth, he finds a logical place to start. In this case, he starts with finding legal words from a hand. He spends time using pen and paper to work out the problem before coding, making sure he has a full understanding of what is going on and how to implement it before actually using the editor.

Sixth, he writes the code for that function. He's not quite satisfied with the function, but he moves on for the time being.

Seventh, he writes tests for the function to see if its performing correctly. These tests will be important as he makes changes to the function later.

Eighth, he refines his find legal words function by using the notion of prefixes (and tests that too.)

Ninth, he finally refines his find legal words function to a form he is satisfied with (and tests it as well.)

Tenth, he takes a pacing break to congratulate himself on his progress.

Then, he repeats steps Five through Ten, finding the next logical place to start by going back to his concept inventory, coding, testing and refining until the program is complete.
